Big picture
The situation, the thesis, why now.
Situation in plain terms

PCI DSS 4.0 made phishing-resistant MFA mandatory in March 2025. Most companies still run SMS OTP or TOTP, which no longer qualify. The obvious replacement is passkeys — free, FIDO2-certified, built into every phone. But passkeys unlock via Face ID or fingerprint, and biometric spoofing via AI-generated faces is a growing attack vector.

AuthenSee replaces the biometric unlock with memory questions. Memory answers unlock a passkey in the device enclave — passkey provides cryptographic strength, memory provides the human proof. Phishing-resistant, biometric-free, satisfies PCI 8.4.2. Built with Aztec's Noir in roughly 3 months. A capable competitor with Noir experience could replicate in 7–9 months once the product is visible.

What's missing right now: a commercial motion. No prospects, no positioning tested, no distribution path. The commercial hire's job is to fix that before the product ships.

The thesis

Passkeys solved the password problem. The next problem — biometric spoofing via AI — is arriving now. AuthenSee solves it before it becomes mainstream pain. The commercial job: find companies already worried about that risk, get early customers signed before the product ships, have paying customers on day one of launch. That converts a science project into a fundable business. The window is 7–9 months before a well-resourced team replicates it.

Why now specifically
Regulation
PCI DSS 4.0 mandatory since March 2025. Phishing-resistant MFA required for all cardholder data environment access. SMS OTP non-compliant. Fines $5k–$100k/month. Budget already allocated.
Technology
Aztec's Noir makes ZK circuits buildable without a cryptography specialist. 3 devs can ship in 3 months. Moat = first-mover + ZK audit + QSA sign-off + Auth0 listing — not the code.
Competition
No shipping product with this architecture exists today. Replication: 7–9 months minimum.
The gap
Product and funding exist. No commercial motion. No prospects. No positioning tested. That is the job.

Product
What AuthenSee actually does and how it's built
Architecture
User sees
Memory questions — draw a secret shape, trace a personal route, pick a symbol. 3–8 seconds. No camera. No fingerprint. Works on any device.
Under the hood
Memory answers unlock a passkey in the device secure enclave. Passkey = 256-bit hardware-generated key. Aztec's Noir generates a ZK proof locally. Server receives "valid/invalid" only — nothing else is transmitted.
Entropy source
The passkey private key (256-bit hardware random), not the memory answers. Memory is the human liveness gate. The entropy is hardware-grade regardless of how memorable the shapes are.
Built with
Aztec's Noir — Rust-like ZK DSL. Abstracts cryptographic complexity, runs in browser via NoirJS. 3 devs can ship in 3 months. A capable competitor with Noir experience could replicate in 7–9 months.
Integration
Auth0 custom Action. 5–10 lines of code. Under 8 engineering hours for any company already on Auth0, Descope, or Stytch.
What it replaces

SMS OTP — phishable, non-compliant under PCI 4.0
TOTP apps — phishable via relay attack
Push notifications — MFA fatigue attacks
Voice OTP — voice synthesis attacks
WebAuthn biometric — biometric spoofing

Why not just use passkeys

Passkeys satisfy PCI 8.4.2 — free, certified, sufficient for compliance. They unlock via Face ID or fingerprint. Biometric spoofing via AI-generated faces is a growing attack vector. AuthenSee removes biometrics from the unlock step. Complementary approaches, not competing ones.

Full release (week 8–9) — for first live pilots on real traffic
ZK proof
Real Noir circuit running via NoirJS + Barretenberg backend. Proof generation under 3 seconds on mid-range Android (Galaxy A series). Client-side only — nothing sent to server except valid/invalid.
Auth0 Action
5–10 lines of code. Documented. Tested on Auth0 tenant. Works as a post-login Action in any Auth0 flow. Should take a developer under 8 engineering hours to integrate.
Passkey binding
Memory answers unlock a passkey stored in device secure enclave (WebAuthn / FIDO2 compliant). 256-bit hardware entropy. Passkey never leaves the device.
Memory normalisation
Shape drawn slightly differently on day 2 still authenticates. Fuzzy matching with defined tolerance. Must be tuned so false reject rate under 5% and false accept rate under 0.01%.
Recovery
At minimum one recovery path: trusted device backup or admin-initiated reset with identity verification. No recovery = no enterprise adoption.
Error states
Wrong memory (graceful retry with limit), timeout, device not supported (fallback to existing MFA), browser incompatibility message.
Browser / device support
Chrome desktop · Safari desktop · Chrome iOS · Safari iOS · Chrome Android. Mid-range Android performance is the critical bar — do not test only on flagship devices.
Pilot logging
Auth attempts · success rate · time-to-complete · drop-off rate · support ticket trigger events. Privacy-preserving — no user data, aggregate only.

PCI DSS 4.0 — what it is

PCI = Payment Card Industry. DSS = Data Security Standard. Rulebook written by Visa, Mastercard, Amex, Discover and JCB for any company touching card data. QSA = Qualified Security Assessor — independent auditor companies hire annually to certify compliance.

Version 4.0 mandatory since March 2025. Key change: phishing-resistant MFA required for all cardholder data environment access. SMS OTP and TOTP no longer qualify. Non-compliance: $5k–$100k/month in fines plus potential loss of card processing rights.

The PCI SSC confirmed synced FIDO2 passkeys satisfy Requirement 8.4.2. AuthenSee's passkey component is FIDO2-compliant. A QSA technical brief is needed to make this case explicitly to a security auditor — without it, the compliance question stalls in every enterprise sales conversation.

PCI DSS 4.0
Global. Mandatory March 2025. Every company handling card payments.
DORA
EU financial sector. In force January 2025. Banks, payment processors, crypto.
NIS2
EU critical infrastructure + digital services. Overlaps with DORA.
GDPR
"Appropriate technical measures." MFA increasingly cited as a baseline expectation.

How MFA gets hacked
Attack vectors by frequency and relevance to AuthenSee's positioning
The key framing: Attackers rarely break MFA directly. They go around it — stealing session tokens after authentication, tricking users into approving fake requests, or intercepting OTPs in transit. Each attack vector maps to a different MFA weakness. Biometric spoofing is real but currently a smaller slice of total MFA failures than social engineering and session theft. The honest story is that AuthenSee addresses one specific and growing vector, not all of them.

1. AiTM — Adversary in the Middle (largest current vector)
How it works
A proxy sits between the user and the real login page. The user sees a perfect replica of the real site, completes login including MFA, but the proxy captures the session token in real time. The attacker then uses that token — no password, no MFA required.
Scale
Token theft accounted for 31% of Microsoft 365 breaches in 2025. Microsoft detected 10,000+ AiTM attacks per month in 2024. AiTM attacks increased 146% year-on-year. Tycoon 2FA — a single phishing-as-a-service kit — sent over 30 million fraudulent emails in one month before being taken down in early 2026.
Cost to attacker
Commercial kits (EvilProxy, EvilGinx, BlackForce) sell on Telegram for $200–$300. No technical skill required.
Does AuthenSee help?
Partially. The ZK proof is domain-bound like a passkey — it won't verify on a fake site. But the session token issued after successful auth is still vulnerable to theft post-authentication. AuthenSee closes the auth step; it doesn't protect the session that follows.
2. MFA fatigue / push bombing
How it works
Attacker has the user's password (from a prior breach). Repeatedly triggers MFA push notifications until the user approves one to make them stop. No technical sophistication required.
Scale
25% of recent attacks involve fraudulent MFA push notifications. The Lapsus$ attacks on Cisco, Microsoft, Okta, and Samsung all used this technique. Cisco confirmed a 2022 breach through this method alone.
Does AuthenSee help?
Yes — fully. AuthenSee has no push notification. There is nothing to bomb. The user completes a memory-based local proof. No approval prompt exists to manipulate.
3. Social engineering / helpdesk impersonation
How it works
Attacker calls IT support impersonating the target user, claims they're locked out, convinces the helpdesk to reset or bypass MFA. AI voice cloning is now making this significantly more convincing — cloning a CEO's voice costs under $10/month via commercial services.
Scale
68% of all breaches involve a human element (Verizon 2024 DBIR). 37% of fraud experts have encountered voice deepfake fraud. A finance worker at Arup was tricked into wiring $25M via a deepfake video conference call in 2024.
Does AuthenSee help?
Partially. AuthenSee doesn't prevent a helpdesk from being socially engineered into resetting an account. But the memory answers are personal and non-transferable in a way that a reset code is not — an attacker who socially engineers a reset still needs to know the user's specific memory patterns.
4. SIM swapping
How it works
Attacker convinces a mobile carrier to transfer the victim's phone number to a SIM they control. All SMS OTPs then go to the attacker. Requires social engineering the carrier, which has become easier with AI voice cloning.
Scale
Primarily affects SMS OTP — the weakest MFA method. Multiple regulators and carriers have moved to deprecate SMS OTP specifically because of this. Under PCI 4.0, SMS OTP no longer qualifies as phishing-resistant.
Does AuthenSee help?
Yes — completely. AuthenSee has no phone number, no SMS, no carrier. Nothing to SIM swap.
5. Biometric spoofing — where AuthenSee is most directly relevant
Current scale
Deepfakes now account for 40% of biometric fraud attempts (Entrust). Biometric attacks surged 340% in 2025. In Jan–Aug 2025, one financial institution alone recorded 8,065 attempts to bypass biometric liveness checks using AI-generated deepfake images. High-quality deepfakes succeeded against 15–30% of commercial facial recognition systems without specialised anti-spoofing in 2025 testing.
Trajectory
Deepfake fraud attempts increased 2,137% over 3 years (2022–2025). Deepfake files grew from 500,000 in 2023 to ~8 million in 2025. Gartner: by 2026, 30% of enterprises will no longer treat standalone face biometric verification as reliable. Cost of a deepfake image to an attacker: $10–$50. Voice clone from 3 seconds of audio: $10/month service.
Honest caveat
Most biometric spoofing today targets KYC onboarding flows (proving identity to a bank), not continuous authentication (Face ID unlocking a passkey). The attack on passkey biometrics specifically is growing but smaller than the headline deepfake numbers suggest. The risk is directionally correct and accelerating — but it is not yet the dominant attack vector on passkeys specifically.
Does AuthenSee help?
Yes — completely. Memory answers cannot be synthesised from a LinkedIn photo or a 3-second voice clip. There is no biometric surface to attack. This is the most direct and defensible use case for AuthenSee.
6. MFA downgrade attacks
How it works
Even companies with passkeys or FIDO2 keys often keep weaker fallback methods (SMS reset, security questions, email link). Attackers force or manipulate the system into offering the fallback. In mid-2025, the PoisonSeed campaign exploited QR code cross-device passkey fallbacks at scale — users scanned a spoofed QR code and unknowingly handed over a valid FIDO assertion. Microsoft and Okta both issued emergency guidance to disable the QR fallback feature.
Does AuthenSee help?
Only if the fallback is also hardened. AuthenSee's memory layer is strong — but if the account recovery flow lets someone reset via email link or SMS, that's the weak point. Recovery flow design is a critical open question for the product.
What this means for AuthenSee's positioning
Strongest claim
Eliminates MFA fatigue attacks entirely (no push prompt exists). Removes biometric attack surface (no Face ID / fingerprint). Is phishing-resistant by design (ZK proof is domain-bound). These are concrete, verifiable, non-marketable claims.
Honest limits
Does not protect the session token after authentication (AiTM still possible post-login). Does not prevent helpdesk social engineering at the account recovery level. Does not address credential stuffing upstream of auth.
The right pitch
Not "AuthenSee stops all MFA attacks." The accurate claim: "AuthenSee closes three specific attack vectors that passkeys and TOTP leave open: push bombing, biometric spoofing, and SIM swapping. It does this without adding biometric attack surface of its own." That is a narrow, true, and defensible claim.

What breaks this model — the four structural risks
Risk 1
Free passkeys absorb the market. If the 30% of biometric-skeptical enterprises simply switch to device-bound passkeys (no biometric unlock required) rather than adding AuthenSee, the SAM collapses. Device-bound passkeys with PIN unlock are already available and free. AuthenSee must demonstrate a meaningful advantage over PIN-unlocked device-bound passkeys specifically.
Risk 2
Liveness detection improves. If biometric liveness detection (anti-spoofing) becomes robust and cheap before AuthenSee gains traction, the deepfake threat to passkeys diminishes and the core purchase rationale weakens. This is an arms race AuthenSee is not competing in.
Risk 3
MAU assumption is wrong. If the average customer is a 50-person fintech with 20k MAUs rather than 100k, year-1 ARR drops to $120k. The model is highly sensitive to this number. Real discovery calls in weeks 4–7 should validate or kill the 100k assumption before scaling outreach.
Risk 4
Replication before traction. If a well-funded competitor ships a Noir-based equivalent by month 7, the SOM window narrows sharply. First-mover only holds if Auth0 marketplace listing and ZK audit are complete — creating a defensible distribution and credibility gap that takes time to replicate, not just code.

The investment thesis does not require the whole market to move. It requires finding the 30% of enterprises already moving away from biometric auth, and converting 35% of those into paying customers. That is a specific, reachable set of buyers — not a bet on a market that does not yet exist.
